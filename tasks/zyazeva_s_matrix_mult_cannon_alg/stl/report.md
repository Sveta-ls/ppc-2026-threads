## 1. Контекст

STL-версия реализует алгоритм Кэннона с использованием исключительно стандартной библиотеки C++ (`std::thread`) без сторонних runtime-библиотек параллелизма. Весь механизм распределения работы между потоками реализован вручную через вспомогательную функцию `ParallelFor`, которая создаёт пул потоков, делит диапазон задач на равные блоки и ожидает завершения каждого потока через `join`.

## 2. Постановка задачи

**Входные данные** (`InType = std::tuple<size_t, std::vector<double>, std::vector<double>>`):

- `sz` — сторона квадратной матрицы (тип `size_t`);
- `m1` — матрица A, хранимая построчно в одномерном `std::vector<double>` размером `sz * sz`;
- `m2` — матрица B, аналогично.

**Выходные данные** (`OutType = std::vector<double>`): результирующая матрица C = A·B, построчно, размер `sz * sz`.

**Ограничения:**

- `sz > 0` — нулевой размер считается недопустимым;
- `m1.size() == sz * sz` и `m2.size() == sz * sz` — несоответствие размеров блокируется в `ValidationImpl`;
- элементы — `double`, переполнение не рассматривается.

**Крайние случаи:**

- матрица 1×1 — корректно обрабатывается основным циклом;
- передача пустых векторов (sz > 0 но size == 0) — отклоняется валидацией.

Число потоков определяется через `std::thread::hardware_concurrency()` в момент создания пула.

## 3. Базовый алгоритм

Структура алгоритма Кэннона идентична OMP-версии: разбиение на `g × g` блоков (где `g = √N`), начальное выравнивание, `g` шагов умножения со сдвигом, сборка результата. Условие применимости: `g > 1 && g*g == n && n % g == 0`, то есть N должно быть полным квадратом. При несоблюдении — fallback к `RegularMultiplication`.

## 4. Схема распараллеливания

`ParallelFor`:**
 
```cpp
// File: stl/src/ops_stl.cpp — функция ParallelFor (анонимное пространство имён)
void ParallelFor(size_t count, const std::function<void(size_t)> &func) {
  size_t threads_count = std::thread::hardware_concurrency();
  if (threads_count == 0) threads_count = 4;
  threads_count = std::min<size_t>(threads_count, count);
 
  std::vector<std::thread> threads(threads_count);
  size_t block_size = count / threads_count;
  size_t remainder  = count % threads_count;
  size_t begin = 0;
 
  for (size_t th = 0; th < threads_count; ++th) {
    size_t end = begin + block_size + (th < remainder ? 1 : 0);
    threads[th] = std::thread([begin, end, &func]() {
      for (size_t i = begin; i < end; ++i) func(i);
    });
    begin = end;
  }
 
  for (auto &th : threads) th.join();
}
```
 
**Карта поток → блок данных:**
 
- Диапазон `[0, count)` делится на `threads_count` частей.
- Первые `remainder` потоков получают `block_size + 1` итераций, остальные — `block_size`. Это обеспечивает равномерное распределение без потери итераций.
- Каждый поток работает со своим непересекающимся поддиапазоном, передаваемым через захват `[begin, end, &func]` (значениями, чтобы избежать race condition на переменных счётчика).
- `join` вызывается для каждого потока после запуска всех: все потоки сначала создаются, затем ожидаются последовательно.
**Синхронизация:**
 
- Барьер реализован через цикл `join` в конце `ParallelFor`. До выхода из функции все потоки завершены.
- Общих накопителей нет: функция `func(i)` для разных `i` пишет в непересекающиеся области памяти (блоки). `atomic` и `mutex` не требуются.
- Между шагами алгоритма Кэннона в `RunImpl` — явный вызов `ParallelFor` завершается (неявный барьер через `join`) до начала следующего шага.
**Параллелизуемые области:**
 
- `InitializeBlocks` — разрезка матриц на блоки;
- `AlignBlocks` — начальное выравнивание;
- `CannonStep` — умножение блоков на каждом шаге;
- `Shift` — циклический сдвиг блоков;
- `Assemble` — сборка результата;
- `RegularMultiplication` (fallback) — параллельный проход по строкам.

## 5. Детали реализации

**Файлы:** `stl/include/ops_stl.hpp`, `stl/src/ops_stl.cpp`
 
Весь вспомогательный код вынесен в анонимное пространство имён. Функция `MulBlock` реализует перемножение одного блока в формате ijk (вращение по строкам `k`), что обеспечивает хорошую локальность кэша при доступе к блоку B.
 
Сдвиг блоков между шагами:
 
```cpp
// File: stl/src/ops_stl.cpp — Shift
void Shift(std::vector<std::vector<double>> &a, std::vector<std::vector<double>> &b,
           std::vector<std::vector<double>> &na, std::vector<std::vector<double>> &nb, size_t g) {
  ParallelFor(g * g, [&](size_t id) {
    size_t i = id / g, j = id % g;
    na[id] = a[(i * g) + ((j + 1) % g)];   // A сдвигается влево
    nb[id] = b[(((i + 1) % g) * g) + j];   // B сдвигается вверх
  });
}
```
 
Записи идут в `na[id]` и `nb[id]` — разные блоки для разных `id`, гонки нет. Чтения из `a` и `b` — только чтение, безопасны. После `Shift` (= после `join`) выполняется `aa = std::move(na)` — однопоточная операция, конкуренции нет.
 
Потенциальная неэффективность: `std::function<void(size_t)>` добавляет overhead на динамическую диспетчеризацию по сравнению с шаблонным лямбда-подходом TBB или compile-time OpenMP директивами.

## 6. Проверка корректности

STL-версия проверялась сравнением с SEQ для всех функциональных тестов, включая размеры, при которых fallback активируется (N не является полным квадратом). Корректность `join`-барьера была дополнительно проверена: ни одна запись в вектор результата не происходит до завершения всех потоков текущего шага. Расхождений не наблюдалось. Оба режима производительности запускались и уложились в лимиты.

STL-версия показала наилучшее ускорение в режиме `task_run` среди OMP, TBB и STL (0.166 с против 0.197 с и 0.214 с). В режиме `pipeline` результаты STL и OMP сопоставимы. Это может объясняться тем, что `std::thread` с явным `join` имеет меньше накладных расходов на синхронизацию в задачах, где все потоки создаются один раз на весь `ParallelFor`.

## 8. Выводы

`std::thread` позволяет реализовать параллельный алгоритм Кэннона без зависимостей от сторонних библиотек. Ручное разбиение диапазонов в `ParallelFor` прозрачно и воспроизводимо, однако создание и уничтожение потоков при каждом вызове `ParallelFor` является потенциальным источником накладных расходов при большом числе шагов.
