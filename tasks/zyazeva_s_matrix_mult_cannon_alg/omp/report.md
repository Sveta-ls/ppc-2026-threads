## 1. Контекст

OpenMP-версия реализует алгоритм Кэннона для блочного умножения матриц, распараллеливая независимые блочные операции между потоками. В отличие от SEQ, где используется простой тройной цикл, здесь матрицы разбиваются на блоки, по сетке grid_size × grid_size, и шаги алгоритма Кэннона выполняются параллельно. При невозможности применить Кэннона (число потоков не является полным квадратом, или размер матрицы не кратен √num_threads) реализация откатывается к параллельному тройному циклу.

## 2. Постановка задачи

**Входные данные** (`InType = std::tuple<size_t, std::vector<double>, std::vector<double>>`):

- `sz` — сторона квадратной матрицы (тип `size_t`);
- `m1` — матрица A, хранимая построчно в одномерном `std::vector<double>` размером `sz * sz`;
- `m2` — матрица B, аналогично.

**Выходные данные** (`OutType = std::vector<double>`): результирующая матрица C = A·B, построчно, размер `sz * sz`.

**Ограничения:**

- `sz > 0` — нулевой размер считается недопустимым;
- `m1.size() == sz * sz` и `m2.size() == sz * sz` — несоответствие размеров блокируется в `ValidationImpl`;
- элементы — `double`, переполнение не рассматривается.

**Крайние случаи:**

- матрица 1×1 — корректно обрабатывается основным циклом;
- передача пустых векторов (sz > 0 но size == 0) — отклоняется валидацией.

## 3. Базовый алгоритм

Алгоритм Кэннона разбивает матрицы A и B на g × g блоков (где g = √num_threads). Начальное смещение (alignment) расставляет блоки A по строкам, а B по столбцам. Затем выполняется g шагов: на каждом шаге блоки перемножаются и сдвигаются (A — влево, B — вверх). Результирующие блоки собираются в итоговую матрицу.
Вспомогательный метод RegularMultiplication применяется как fallback при невозможности применить Кэннона.

## 4. Схема распараллеливания

Параллелизуемые области:

AlignBlocks — начальное выравнивание блоков (двойной цикл по блокам сетки);
CannonStep — умножение и сдвиг блоков на каждом шаге (двойной цикл по блокам);
AssembleResult — сборка итоговой матрицы из блоков;
RegularMultiplication (fallback) — цикл по строкам результирующей матрицы.

cpp// File: omp/src/ops_omp.cpp — AlignBlocks

# pragma omp parallel for default(none) \

    shared(blocks_a, blocks_b, aligned_a, aligned_b, grid_size, grid_size_t) \
    collapse(2)
for (int i = 0; i < grid_size; ++i) {
  for (int j = 0; j < grid_size; ++j) {
    // aligned_a[block_idx] = blocks_a[...]; (чтение из различных блоков)
    // aligned_b[block_idx] = blocks_b[...]; (чтение из различных блоков)
  }
}
Атрибуты переменных:

shared: входные и выходные блочные векторы (blocks_a, blocks_b, aligned_a, aligned_b, blocks_c, res_m), размерные константы (grid_size, block_size и их size_t-аналоги);
private: итераторы цикла i и j неявно private при #pragma omp parallel for;
reduction не используется явно, поскольку каждый поток пишет в свой блок результата — гонки нет (блоки не перекрываются).

collapse(2) применён к двойному циклу по grid_size × grid_size, что позволяет OpenMP распределять все grid_size² итераций между потоками, а не только строки внешнего цикла. Это важно при малом grid_size, когда без collapse число независимых итераций внешнего цикла совпадает с числом потоков или меньше.
schedule: явно не задан → используется schedule(static) по умолчанию. Нагрузка равномерна (блоки одинакового размера), поэтому статическое расписание оптимально.
Барьеры: в конце каждой #pragma omp parallel for стоит неявный барьер. Между шагами алгоритма Кэннона (for (int step = 0; step < grid_size; ++step)) барьер выполняется неявно при завершении каждого CannonStep, что корректно: следующий шаг не может начаться раньше, чем все блоки текущего шага перемножены и сдвинуты.

## 5. Детали реализации

Файлы: omp/include/ops_omp.hpp, omp/src/ops_omp.cpp
Ключевое отличие от SEQ: RunImpl определяет доступность алгоритма Кэннона проверкой IsPerfectSquare(num_threads) && sz >= num_threads && (sz % (int)sqrt(num_threads) == 0). Это гарантирует, что блочная сетка делит матрицу ровно.
cpp// File: omp/src/ops_omp.cpp — CannonStep

# pragma omp parallel for default(none) \

    shared(aligned_a, aligned_b, blocks_c, grid_size, block_size, grid_size_t) \
    collapse(2)
for (int i = 0; i < grid_size; ++i) {
  for (int j = 0; j < grid_size; ++j) {
    const size_t block_idx = (static_cast<size_t>(i) *grid_size_t) + static_cast<size_t>(j);
    MultiplyBlocks(aligned_a[block_idx], aligned_b[block_idx], blocks_c[block_idx], block_size);
  }
}
Каждый поток работает со своим block_idx, записывая в blocks_c[block_idx]. Сдвиг блоков после умножения также выполняется параллельно: новые aligned_a и aligned_b создаются в локальных new_aligned_* и только после завершения всего параллельного блока заменяют старые (неявный барьер).

## 6. Проверка корректности

OMP-версия проверялась сравнением с SEQ для всех функциональных тестов, включая матрицы 1×1, 2×2 и нескольких средних размеров. Ветка RegularMultiplication (fallback) тестировалась при числе потоков 1 и при нечётных значениях. Расхождений не наблюдалось. Дополнительно проверялась устойчивость результата при PPC_NUM_THREADS = 1, 2, 4.

## 8. Выводы

OpenMP дал ускорение в 2.6–4.1× на тестовом окружении. Ключевым фактором является параллельная обработка независимых блоков алгоритма Кэннона. Ограничения: число потоков должно быть полным квадратом и делителем размера матрицы; при несоблюдении условий код откатывается к `RegularMultiplication` без потери корректности. Overhead на сдвиг блоков (аллокация `new_aligned_*` на каждом шаге) ограничивает дальнейшее масштабирование.
