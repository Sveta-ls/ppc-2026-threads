## 1. Контекст

OpenMP-версия реализует алгоритм Кэннона для блочного умножения матриц,
распараллеливая независимые блочные операции между потоками.

В отличие от SEQ, где используется простой тройной цикл,
здесь матрицы разбиваются на блоки по сетке `grid_size × grid_size`,
а шаги алгоритма Кэннона выполняются параллельно.

При невозможности применить Кэннона
(число потоков не является полным квадратом
или размер матрицы не кратен `√num_threads`)
реализация откатывается к параллельному тройному циклу.

## 2. Постановка задачи

**Входные данные**
(`InType = std::tuple<size_t, std::vector<double>, std::vector<double>>`):

- `sz` — сторона квадратной матрицы
  типа `size_t`;
- `m1` — матрица `A`,
  хранимая построчно
  в одномерном `std::vector<double>`
  размером `sz * sz`;
- `m2` — матрица `B`,
  аналогично.

**Выходные данные**
(`OutType = std::vector<double>`):

результирующая матрица `C = A·B`, хранимая построчно, размер `sz * sz`.

**Ограничения:**

- `sz > 0` —
  нулевой размер считается недопустимым;
- `m1.size() == sz * sz`
  и `m2.size() == sz * sz` —
  несоответствие размеров блокируется
  в `ValidationImpl`;
- элементы имеют тип `double`,
  переполнение не рассматривается.

**Крайние случаи:**

- матрица `1×1` —
  корректно обрабатывается основным циклом;
- передача пустых векторов
  (`sz > 0`, но `size == 0`) —
  отклоняется валидацией.

## 3. Базовый алгоритм

Алгоритм Кэннона разбивает матрицы `A` и `B`
на `g × g` блоков,
где `g = √num_threads`.

Начальное смещение (`alignment`)
расставляет блоки:

- `A` — по строкам;
- `B` — по столбцам.

Затем выполняется `g` шагов:

- блоки перемножаются;
- `A` циклически сдвигается влево;
- `B` циклически сдвигается вверх.

После завершения всех шагов результирующие блоки
собираются в итоговую матрицу.

Вспомогательный метод `RegularMultiplication`
используется как fallback, если условия применения Кэннона
не выполняются.

## 4. Схема распараллеливания

Параллелизуемые области:

- `AlignBlocks` —
  начальное выравнивание блоков;
- `CannonStep` —
  умножение и сдвиг блоков
  на каждом шаге;
- `AssembleResult` —
  сборка итоговой матрицы;
- `RegularMultiplication` —
  fallback-режим
  с параллельным циклом по строкам.

Пример распараллеливания
начального выравнивания блоков:

```cpp id="v2f0n1"
// File: omp/src/ops_omp.cpp — AlignBlocks
#pragma omp parallel for default(none) \
    shared(blocks_a, blocks_b, aligned_a, aligned_b, \
           grid_size, grid_size_t) \
    collapse(2)
for (int i = 0; i < grid_size; ++i) {
  for (int j = 0; j < grid_size; ++j) {
    // aligned_a[block_idx] = blocks_a[...];
    // aligned_b[block_idx] = blocks_b[...];
  }
}

Атрибуты переменных:

shared:
входные и выходные блочные векторы (blocks_a, blocks_b,
aligned_a, aligned_b, blocks_c, res_m);

shared:
размерные параметры (grid_size, block_size
и их size_t-аналоги); 

private:
итераторы i и j являются private,
так как используются внутри parallel for.

reduction не используется, поскольку каждый поток
записывает данные в собственный блок результата,
а блоки не пересекаются.

Collapse

collapse(2) применяется к двойному циклу
по сетке grid_size × grid_size.

Это позволяет OpenMP распределять между потоками все grid_size² итераций,
а не только итерации внешнего цикла.

Такой подход особенно важен при небольшом grid_size, когда без collapse
число независимых задач может оказаться меньше, чем число потоков.

Schedule

Явное расписание не задаётся, поэтому используется schedule(static) по умолчанию.

Нагрузка равномерна, так как все блоки имеют одинаковый размер,
поэтому статическое распределение является оптимальным.

Барьеры

В конце каждого #pragma omp parallel for существует неявный барьер.

Между шагами алгоритма Кэннона барьер также необходим,
поскольку следующий шаг не должен начинаться,
пока все блоки текущего шага не завершили умножение и сдвиг.

5. Детали реализации

Файлы:

omp/include/ops_omp.hpp
omp/src/ops_omp.cpp

Ключевое отличие от SEQ:

RunImpl определяет, можно ли применять алгоритм Кэннона.

Проверяется условие:

IsPerfectSquare(num_threads) &&
sz >= num_threads &&
(sz % static_cast<int>(std::sqrt(num_threads)) == 0)

Это гарантирует, что блочная сетка делит матрицу без остатка.

Пример параллельного шага Кэннона:

// File: omp/src/ops_omp.cpp — CannonStep
#pragma omp parallel for default(none) \
    shared(aligned_a, aligned_b, blocks_c, \
           grid_size, block_size, grid_size_t) \
    collapse(2)
for (int i = 0; i < grid_size; ++i) {
  for (int j = 0; j < grid_size; ++j) {
    const size_t block_idx =
        (static_cast<size_t>(i) * grid_size_t) +
        static_cast<size_t>(j);

    MultiplyBlocks(
        aligned_a[block_idx],
        aligned_b[block_idx],
        blocks_c[block_idx],
        block_size);
  }
}

Каждый поток работает со своим block_idx
и записывает данные в blocks_c[block_idx].

Гонок данных не возникает, так как блоки результата не пересекаются.

Сдвиг блоков после умножения также выполняется параллельно:

создаются новые new_aligned_a и new_aligned_b;
после завершения параллельного участка старые контейнеры заменяются новыми.

Корректность синхронизации обеспечивается неявным барьером OpenMP.

число потоков должно быть полным квадратом; размер матрицы должен делиться на размер сетки блоков.
