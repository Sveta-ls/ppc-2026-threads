## 1. Введение
Задача — произведение двух квадратных вещественных матриц A и B размера N×N. Алгоритм Кэннона является примером блочного параллельного умножения матриц: он разбивает матрицы на блоки, назначает каждый блок отдельной рабочей единице и на каждом шаге выполняет локальное умножение с циклическим сдвигом блоков.
 
Пять реализаций — SEQ, OMP, TBB, STL, ALL — применяют одну и ту же вычислительную схему с разными механизмами распараллеливания, что позволяет честно сравнивать именно накладные расходы и выигрыш каждой технологии, а не разные алгоритмы.

## 2. Единая постановка задачи
**Входные данные** (`InType = std::tuple<size_t, std::vector<double>, std::vector<double>>`):
 
- `sz` — сторона квадратной матрицы (`size_t`);
- `m1` — матрица A, построчно, размер `sz × sz`;
- `m2` — матрица B, построчно, размер `sz × sz`.
**Выходные данные** (`OutType = std::vector<double>`): матрица C = A·B, построчно, размер `sz × sz`.
 
**Ограничения:** `sz > 0`; `m1.size() == sz * sz`; `m2.size() == sz * sz`. Проверяются в `ValidationImpl` каждого backend-а.
 
**Критерий корректности:** результат совпадает с результатом SEQ-реализации на тех же входных данных (поэлементное сравнение с допуском на погрешность `double`).
 
**Ограничения применимости алгоритма Кэннона:**
 
| Technology | Условие применения Кэннона | Fallback |
|---|---|---|
| SEQ | — | тройной цикл (всегда) |
| OMP | `√num_threads` — целое, `sz % √num_threads == 0` | `RegularMultiplication` (OMP) |
| TBB | `sz % FindGridSize(sz) == 0` (наилучший делитель) | сетка 1×1 |
| STL | `g = √sz` — целое, `g > 1`, `sz % g == 0` | `RegularMultiplication` (STL) |
| ALL | `√mpi_size_` — целое, `sz % √mpi_size_ == 0` | `RegularMultiplication` (OMP, ранг 0) |

**Переменные окружения:** `PPC_NUM_THREADS` (= `OMP_NUM_THREADS` для OMP- и ALL-версий), `PPC_NUM_PROC` (число MPI-рангов для ALL). Автоматически 16 потоков.
 
**Генерация данных:** входные матрицы генерируются тестовой инфраструктурой курса (`MakeAllPerfTasks`); конкретный размер N фиксируется runner-ом.
 
**Формула ускорения:**
```
Speedup = T_seq / T_backend
```
где `T_seq` — время SEQ-версии в соответствующем режиме (`pipeline` или `task_run`).
 
**Формула эффективности:**
```
Efficiency = Speedup / workers
```
где `workers` — число рабочих единиц: для OMP/TBB/STL — число потоков; для ALL — `ranks × threads_per_rank`.
 
**Агрегирование:** приведены единичные значения из одного запуска тестовой инфраструктуры курса. Для повышения воспроизводимости рекомендуется выполнить 3–5 повторов и использовать медиану; дополнительная стабилизация (governor `performance`, `taskset`) при данном запуске не применялась.
 
**Режимы тестирования:** `pipeline` (тест всего пайплайна `Validation → PreProcessing → Run → PostProcessing`) и `task_run` (тест только фазы `Run`). Результаты двух режимов не смешиваются в одной строке таблицы.

Все пять backend-ов сравнивались с SEQ на едином наборе функциональных тестов из `tests/functional/main.cpp`. Тесты включают:

## 6. Интерпретация различий
**SEQ** реализует классический тройной цикл O(N³) без каких-либо оптимизаций. Доступ к столбцу матрицы B является неудобным для кэша, поэтому SEQ-время систематически хуже, чем можно было бы получить при блочной реорганизации данных. Именно это делает baseline «честным»: параллельные версии выигрывают не только от многопоточности, но и частично от блочной структуры алгоритма Кэннона.
 
**OMP** показывает ускорение 4.09× / 2.62×. Распараллеливание через `#pragma omp parallel for collapse(2)` покрывает инициализацию блоков, выравнивание, Cannon-шаги и сборку результата. Ограничение: между шагами Кэннона аллоцируются новые векторы блоков (`new_aligned_a`, `new_aligned_b`), что создаёт накладные расходы на память и снижает ускорение в режиме `task_run`. В режиме `pipeline` overhead компенсируется; разница между `pipeline` и `task_run` (4.09× vs 2.62×) объясняется, вероятно, различиями в условиях кэша при повторных запусках.
 
**TBB** показывает результаты, близкие к OMP (3.78× / 2.85×). Ключевое отличие — использование индексных карт (`map_a`, `map_b`) вместо физического копирования блоков, что снижает объём аллокаций на каждом шаге. `cache_aligned_allocator` уменьшает ложное разделение кэша. В режиме `task_run` TBB незначительно опережает OMP (0.197 с против 0.214 с), что согласуется с меньшим объёмом аллоцируемой памяти на шаг.
 
**STL** показывает наилучший результат среди однопроцессных реализаций в режиме `task_run` (3.38×, 0.166 с). `ParallelFor` создаёт потоки один раз на каждую параллельную операцию и синхронизирует их через `join`, что при равномерной нагрузке оказывается эффективным. В режиме `pipeline` результаты STL и OMP практически совпадают (0.173 с против 0.174 с). Потенциальный источник накладных расходов — использование `std::function` (dynamic dispatch) и повторное создание потоков при каждом вызове `ParallelFor`.
 
**ALL (MPI + OMP)** даёт ускорение 9.89× / 8.33× — значительно выше любой однопроцессной реализации. Двухуровневый параллелизм: MPI распределяет блоки по процессам (каждый процесс владеет одним блоком и выполняет `grid` шагов умножения со сдвигом), OMP ускоряет вспомогательные операции внутри ранга. Коммуникационные издержки на `MPI_Bcast` полной матрицы при старте и `MPI_Sendrecv_replace` на каждом шаге существуют, однако при данном размере задачи вычислительный выигрыш их перекрывает. Последовательная рассылка блоков через `MPI_Send` на ранге 0 является архитектурным ограничением при большом числе процессов.
 
Близость результатов OMP, TBB и STL в режиме `pipeline` (0.173–0.189 с) объясняется тем, что при данном числе потоков все три реализации упираются в одно и то же ограничение — размер сетки блоков `g × g` и объём памяти, необходимый для хранения промежуточных блоков. Дальнейшее увеличение числа потоков без увеличения N не даст линейного прироста.

## 8. Заключение
Для задачи умножения матриц алгоритмом Кэннона лучшим по абсолютной производительности является гибридная версия ALL (MPI + OMP) с ускорением ~9.9× / ~8.3×. Среди однопроцессных реализаций STL показала лучший результат в режиме `task_run` (3.38×), OMP и STL — сопоставимые результаты в режиме `pipeline` (4.09× / 4.12×).
 
Выбор технологии зависит от условий эксплуатации: при наличии нескольких MPI-процессов ALL является очевидным выбором; при однопроцессном запуске STL или OMP практически равнозначны, а TBB уступает им незначительно.
